# Frontend Design

React 主播团队工作台 PC 前端，已对齐后端 JWT 鉴权和统一响应包装。

## 页面能力

- Home 产品落地页：根据 PDF 课题背景展示直播竞拍闭环和主播团队工作台入口；
- Login 独立工作台登录页：只保留主播主账号/团队子账号登录，登录后进入 `/host` 或 `?next=` 指定后台路径；
- 主播/运营/admin 登录后创建、开拍、揭示信任卡、Duel、落锤、异常取消；
- 后台指标卡、拍品表格、信任揭示、危险操作、契约待扩展禁用态；
- 业务错误展示来自 `reply.result.message`。

当前项目不实现用户 H5 / 观众端 / 小程序入口或页面；这些端后续另起独立项目。

## 后端连接

默认同源代理，部署时可覆盖：

```text
VITE_API_BASE=
VITE_WS_BASE=ws://当前前端 host
```

真实后端默认端口：`18080`。

## Home 冻结状态

2026-05-21 用户确认 Home 页设计完成并冻结。当前最终方案：浅蓝/粉梦幻手绘玻璃拟态风格，随机装饰元素避开主卡，主卡内单按钮“进入后台”，无顶部入口、无用户 H5 入口、无拖拽调试工具。后续不要主动改 Home 视觉，除非用户明确要求继续调整。

## 当前前端项目设计边界

当前 `live-auction-bid-frontend` 是 **主播团队工作台 PC Web**，不是“后台 + 用户 H5”的双入口项目。

- **当前项目只做主播团队工作台**：拍品准备、竞拍玩法配置、开拍控场、异常取消、成交处理、数据看板、实时链路诊断和团队子账号权限，PC 工作台优先。一个主播主账号固定绑定一个直播间，可创建多个团队子账号协作；发布竞拍时直播间固定展示，不允许选择多个直播间，也不再选择“主播”，只选择本场负责人/授权子账号。
- **用户 H5 竞拍端另起项目**：移动端优先，承担用户实时出价、排名/倒计时同步、提醒、订单结果与模拟支付。
- **小程序端也是独立端**：若后续需要，按微信/抖音小程序规范单独开发，复用后端 API/WebSocket。

因此 Home 页不再放“用户 H5 竞拍端 + 主播团队工作台”两个同级入口，而只保留进入主播团队工作台的入口；H5/小程序作为系统架构中的独立端记录在文档中。

## Home / Login 视觉取舍

用户指出前端不能直接进入工作台，应参考 Sub2API 的 `/home` 与 `/login`。当前公开入口层：

- **Home 页**：已冻结为浅蓝/粉、梦幻、手绘、玻璃拟态风格，只保留进入后台入口。
- **Login 页**：采用与 Home 同方向的柔和蓝粉玻璃卡片；只做账号登录，账号所属主播空间和团队权限由系统识别，不在登录页选择主账号/子账号角色。
- **架构分层**：`/home` 负责说明当前后台产品，`/login` 负责后台身份，`/host`/`/admin` 负责主播团队工作台。用户 H5/小程序端作为后续独立项目，不在当前前端中提供入口。
- **前端 mock 仅作 UI 原型过渡**：Login 仍调用真实 `login`；当前后台为快速验证 PC 工作台信息架构和视觉，可短期使用本地 mock 数据展示竞拍、商品、订单、WebSocket、风控、权限角色等页面。接真实接口时必须删除所有 mock 痕迹，不允许 mock 与真实 API/WebSocket 并存。

## 主播团队工作台视觉取舍

主播团队工作台按 **PC 工作台 + 直播间中控台台** 设计，而不是移动观众端的沉浸式直播间：

- **整体视觉**：延续 Home 的浅蓝/粉、梦幻、手绘玻璃方向，但不修改已冻结 Home 文件。
- **状态总览**：把拍品总数、直播中、草稿、成交/取消、GMV 前置，满足后台“先看系统状态，再操作”的信息架构。
- **当前拍品为主工作台**：当前价、状态、版本号、结束时间、领先会员、阶段、防狙击规则集中展示；开拍/进入 Duel/落锤按钮启用条件直接反映后端状态机。
- **创建拍品常驻**：作为后台常驻工具，不抢当前控场台的视觉优先级；仍然只调用真实 `CreateLot`。
- **信任揭示独立分区**：卡片揭示是直播成交信任链路的一部分，和价格控制区拆开，避免主播误操作。
- **危险操作分区**：异常取消放入 Danger Zone，使用原因输入 + 红色 danger 按钮，并仅对 `LOT_STATUS_LIVE` 启用；当前已接入真实 `POST /api/lots/{lot_id}/cancel` 契约。
- **响应式保底**：大屏使用侧边栏 + 主工作台；中小屏折叠为单列，保障演示不崩。

本轮没有引入大型 UI 库；只在 `src/app/styles.css` 追加后台/Login 专用 CSS，避免触碰 Home 视觉规则。

## 设计取舍

- 当前不引入全局状态库，认证状态用一个轻量 Auth Token Adapter 管理；等页面复杂度提升后再考虑 Zustand/Redux。
- 不引入 OpenAPI 生成文件，因为后端当前没有最新 OpenAPI artifact；避免 stale schema 误导。
- 主播团队工作台最终必须接真实后端 API/WebSocket。当前阶段允许页面内 mock 作为 UI 原型过渡；一旦对应契约落地，必须删除 mock 数据源、mock 动画/fallback 和相关注释，改为真实 service/adapter 数据，不允许两套实现并存。

## Development Rule

- 前端 UI 原型阶段可以使用页面内本地 mock 数据先完成视觉、布局和交互验证。
- mock 必须是过渡态：不得封装成长期 mock service，不得冒充真实接口，不得作为最终交付依据。
- 接真实后端 API/WebSocket 时必须删除所有 mock 痕迹，包括 mock 数组、mock 类型、mock 动画、fallback 文案和相关注释。
- 最终前端数据源必须来自真实 API/WebSocket contract 或外部测试环境。

## 本场拍品队列页设计补充（2026-05-21）

`/admin/auctions` 与 `/host/auctions` 作为“当前固定直播间的竞拍队列与竞拍状态管理页”，不是通用 SaaS 表格页。页面不提供直播间切换或直播间筛选；所有筛选、当前竞拍、下一场竞拍、WebSocket 状态和操作都围绕当前主播空间固定绑定的直播间展开。

第一版优先读取真实 `GET /api/lots?room_id={currentRoomId}`、`GET /api/rooms/{currentRoomId}/snapshot`，并使用现有 start / settle / cancel 接口执行操作。订单、日志、排期等接口未接入处只显示“待接入”，不构造长期 mock 数据。

## 实时链路诊断设计补充（2026-05-21）

原“实时同步状态”已收敛为“实时链路诊断 / 直播间链路监控”。该能力只围绕当前主播空间唯一直播间，不做跨直播间运维大屏，也不展示前端无法计算的全服连接数、广播成功率、Redis 全局指标等。

P0 能力放入业务页：直播间中控台页 `SystemHealthCard` 和本场拍品队列页顶部同步状态胶囊展示连接状态、平均延迟、最近心跳、重连次数、服务器时间偏移、最近事件类型和事件序号，并提供重新同步房间快照与事件日志入口。独立 `/admin/realtime` 作为 P1 “实时链路诊断”页，仅展示当前直播间信息、客户端可计算指标、最近事件流和异常诊断。

## LiveAuction Studio 产品定位（2026-05-21）

后台统一定位为“主播团队工作台 / LiveAuction Studio / 直播间竞拍工作台”，不是平台级管理后台，也不是通用 SaaS 控制台。一个主播主账号只有一个固定直播间，子账号角色为场控、商品助理、订单客服、数据复盘，所有功能都围绕该直播间的直播前准备、直播中控场和直播后成交处理展开。

页面命名统一：添加拍品/创建竞拍拍品、本场拍品队列/今日排品、直播间中控台、成交处理、实时同步状态/实时链路诊断。默认称呼使用主播、场控、商品助理、订单客服，不使用平台管理者作为默认用户口吻。
