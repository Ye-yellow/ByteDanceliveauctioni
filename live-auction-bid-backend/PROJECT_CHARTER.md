# liveauction 项目最强纲领

> 最高需求基线：用户 2026-05-20 提供的《抖音电商AI全栈课题-直播竞拍全栈系统（宣讲版）》PDF。
>
> 本文件优先级高于普通 README、临时计划和局部实现说明。后续产品、架构、后端、前端、测试、答辩文档都必须先对齐这里，再展开实现。

## 1. 不再自由发挥，按课题做

liveauction 是“抖音电商 AI 全栈课题：直播竞拍全栈系统”的实现项目，不是泛泛的拍卖 demo。

所有功能取舍必须服务课题评分点：

- WebSocket 实时通信完整实践；
- 高并发出价下的数据一致性；
- 状态机设计与复杂竞拍规则；
- React + TypeScript + WebSocket 前端交互；
- Go + MySQL + Redis 后端主路径；
- 从数据库设计到前端交互的一套全栈闭环；
- AI 工具使用流程和人工关键决策点沉淀。

## 2. 必须闭合的业务主流程

```text
商品上架
  -> 规则配置
  -> 主播开拍
  -> 用户实时出价
  -> WebSocket 同步价格/排名/倒计时
  -> 自动延时 / 封顶价自动成交 / 主播异常取消
  -> 竞拍结束
  -> 自动生成订单
  -> 用户查看结果 / 模拟支付
```

如果一个实现不能推进这条链路，必须说明它服务哪个评分点；否则不做。

## 3. P0 评分硬点

以下能力是 P0，不是加分项：

1. 0 元起拍；
2. 固定加价幅度；
3. 封顶价自动成交；
4. 结束前自动延时 10-30 秒；
5. 主播异常取消；
6. 成交后自动生成订单；
7. 商品管理：状态、进度、成交结果；
8. 用户端：移动 H5/直播间体验；
9. 用户实时出价、排名、被超越/领先/延时/结束提醒；
10. WebSocket 心跳保活、自动重连、断线后快照恢复；
11. 100+ 并发出价时价格、排名、订单一致；
12. 答辩材料中能逐项对应课题要求。

## 4. 技术主线

- Frontend：React + TypeScript + WebSocket，移动端优先但兼容 PC 演示；
- Backend：Go + Kratos + Proto；
- Data：MySQL 8 + GORM 存核心业务；
- Realtime/Cache：Redis 7 用于高频读写、幂等、Stream/outbox；
- Registry：Consul；
- Local Demo：Docker Compose 一键启动；
- Contract：Proto 为源头，前端通过契约防腐层对齐。

## 5. 工程铁律

- 禁止 demo/mock 思路；每次开发都按系统开发推进；
- 被正式方案替代的旧实现直接删除，不做 fallback、不并存；
- 业务错误统一包装到 `reply.result`，不要 `reply + error` 双语义；
- 跨层业务测试放到 `app/auction/service/test`，HTTP 黑盒契约测试放到 `test/e2e`，不污染 internal 实现目录；
- 每个基础设施引入、每次设计变更都必须写工程思考、设计模式、边界和后续演进；
- data 层持有 GORM/Redis/事务/cache/outbox，server 层持有 transport/registry，biz 层只放业务规则和接口；
- 有 `ref/` 的包先参考 ref 风格，不自由发挥；
- 先完成课题硬闭环，再做炫技和加分。

## 6. 当前最近实施路线

1. 后端补 `BidRule.cap_price`，实现封顶价自动成交；
2. 后端补 `CancelLot`，支持主播异常取消；
3. 后端补订单聚合和订单表，成交时事务内生成订单；
4. 自动延时事件进入 WebSocket，并在前端提醒；
5. 前端补 WebSocket 重连、心跳、快照恢复；
6. 前端改移动 H5 观众体验和提醒 feed；
7. 补压测、状态机图、时序图、评分点对照清单。
