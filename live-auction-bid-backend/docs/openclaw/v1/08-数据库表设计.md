# 08-数据库表设计

> 状态：V1 第一版库表设计  
> 适用范围：直播竞拍 V1 后端 MySQL / Redis 数据模型  
> 原则：先服务单直播间 demo 闭环，但表结构要能自然演进到多直播间、高并发、事件可追溯。

## 1. 设计目标

V1 数据库不只是把 proto JSON 塞进一张表，而是要表达清楚直播竞拍核心业务对象：

- 拍品聚合；
- 竞拍规则；
- 信任揭示卡片；
- 出价流水；
- 事件日志；
- 幂等控制。

其中 MySQL 负责可追溯、可查询、可恢复的数据；Redis 负责高频、短路径、低延迟的数据。

## 2. 存储边界

### 2.1 MySQL

MySQL 保存权威业务数据：

```text
auction_lots          拍品聚合主表
action_trust_cards    信任揭示卡片表
auction_bids          出价流水表
auction_events        竞拍事件日志表
```

> 注：实际表名为 `auction_trust_cards`，不要写成 action。

### 2.2 Redis

Redis 保存高频运行态数据：

```text
auction:idem:{lot_id}:{idempotency_key}  出价幂等键
auction:ranking:{lot_id}                 后续排行榜 ZSET 预留
auction:snapshot:{room_id}               后续房间快照缓存预留
auction:events:{room_id}                 后续事件流预留
```

V1 当前只落地幂等键；排行榜仍可由 MySQL 出价流水计算，后续再替换为 Redis ZSET。

## 3. 表关系

```text
auction_lots 1 ─── n auction_trust_cards
auction_lots 1 ─── n auction_bids
auction_lots 1 ─── n auction_events
```

`room_id` 是直播间维度索引，V1 可以固定为 demo，但表结构必须保留。

## 4. auction_lots：拍品聚合主表

### 4.1 作用

保存拍品当前权威状态。服务重启后，直播间快照可以从这里恢复。

| image_url | varchar(1024) | 主图 URL |
| status | int | LotStatus 枚举值 |
| start_price_amount | bigint | 起拍价，最小货币单位 |

| min_increment_currency | varchar(16) | 最低加价币种 |
| duration_seconds | int | 竞拍总时长 |

## 7. auction_events：竞拍事件日志表

### 7.1 作用

保存关键事件，用于后续复盘、答辩展示、问题排查和 WebSocket 重放。

V1 代码可以先只广播事件，不强制落库；但表结构先设计好。

### 7.2 字段设计

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | varchar(64) | 事件 ID，主键 |
| room_id | varchar(64) | 直播间 ID |
| lot_id | varchar(64) | 拍品 ID |
| type | int | AuctionEventType 枚举值 |
| occurred_at_unix_ms | bigint | 事件发生时间 |
| reason | varchar(512) | 失败原因，例如 BID_REJECTED |

```text

```

## 8. Redis Key 设计

### 8.1 出价幂等键

```text
auction:idem:{lot_id}:{idempotency_key}
```

Value：`Bid` 的 proto JSON。

用途：同一次出价请求重试时返回同一条 bid，不重复入账。

### 8.2 排行榜预留

```text
auction:ranking:{lot_id}
```

类型：ZSET。

Score：出价金额。  
Member：用户 ID 或用户快照 JSON。

V1 当前仍从 MySQL `auction_bids` 计算排行榜，避免过早引入 Redis 排行榜一致性问题。

### 8.3 快照缓存预留

```text
auction:snapshot:{room_id}
```

类型：String JSON。

用于高频进入直播间时快速恢复房间状态。

## 9. 为什么仍保留 payload JSON

V1 表里同时有结构化字段和 `payload JSON`，原因：

- 结构化字段用于查询、筛选、排序；
- payload 用于快速恢复 proto 完整对象；
- proto 迭代时，V1 可以减少大量字段拆表成本；
- 后续稳定后再逐步把核心字段完全结构化。

这不是“偷懒只塞 JSON”，而是 V1 阶段的演进式设计。
