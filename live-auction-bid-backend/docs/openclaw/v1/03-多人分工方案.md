# V1 多人分工方案

> 状态：第一版分工草案，用于团队协作讨论。  
> 原则：先按职责边界拆，再按 GitHub issue/分支落地。  
> 注意：当前阶段先做需求和设计，不急着抢实现。

## 1. 分工原则

这个项目不要简单分成“一个人前端、一个人后端”。

直播竞拍系统涉及：

- 交易状态机；
- 实时通信；
- 前端直播间交互；
- 主播运营台；
- 玩法机制；
- 测试与答辩材料。

所以 V1 建议按模块分工：

```text
产品需求负责人
系统架构负责人
后端交易核心负责人
后端实时链路负责人
前端观众端负责人
前端主播端负责人
测试与答辩负责人
```

如果人数不够，可以一人兼多个角色；如果人数更多，再继续拆。

## 2. 推荐角色划分

## 2.1 产品需求负责人

### 负责内容

- 梳理赛题要求；
- 明确 V1 范围；
- 定义用户角色；
- 定义业务流程；
- 维护需求文档；
- 控制需求不要无限膨胀。

### 主要产出

```text
docs/openclaw/v1/01-需求草案.md
```

后续可以补：

```text
docs/openclaw/v1/04-用户故事.md
docs/openclaw/v1/05-验收标准.md
```

### 需要确认的问题

- V1 是否单直播间；
- V1 是否游客出价；
- 主播端做到什么程度；
- 信任揭示和 Duel Mode 是否作为 V1 核心亮点。

---

## 2.2 系统架构负责人

### 负责内容

- 设计整体系统架构；
- 明确模块边界；
- 定义数据流、事件流；
- 定义未来可扩展方向；
- 保证系统有“大厂风格”：状态机、事件驱动、可追溯、高并发演进方案。

### 主要产出

```text
docs/openclaw/v1/02-系统设计构思.md
api/auction/service/v1/auction.proto
```

### 重点关注

- 竞拍状态机；
- WebSocket 实时广播；
- 出价原子性；
- 事件日志；
- Redis/MySQL 后续演进；
- AI 不进入交易核心链路。

---

## 2.3 后端交易核心负责人

### 负责内容

负责竞拍最核心的业务规则。

模块位置：

```text
app/auction/service/internal/biz/
```

### 主要对象

```text
Lot
Bid
BidRule
AuctionSession
Settlement
RankingItem
TrustRevealCard
DuelState
```

### 核心逻辑

- 创建拍品；
- 开拍；
- 出价校验；
- 当前价更新；
- 领先者更新；
- 反狙击延时；
- 落锤成交；
- 状态机非法操作拒绝。

### 验收标准

必须能用单元测试证明：

- DRAFT 不能出价；
- LIVE 可以合法出价；
- 低于最低加价的出价被拒；
- SETTLED 后不能继续出价；
- 最后 N 秒出价会触发延时；
- 落锤后 winner/final_price 正确。

---

## 2.4 后端实时链路负责人

### 负责内容

负责直播间 WebSocket 和事件广播。

模块位置：

```text
app/auction/service/internal/realtime/
app/auction/service/internal/server/
```

### 核心能力

- 房间连接管理；
- 用户进入/离开；
- 直播间订阅；
- 广播 AuctionEvent；
- 连接断开清理；
- 重连后快照恢复。

### 主要事件

```text
ROOM_SNAPSHOT
LOT_STARTED
BID_ACCEPTED
BID_REJECTED
RANKING_UPDATED
TRUST_REVEALED
DUEL_STARTED
LOT_SETTLED
```

### 验收标准

- 两个浏览器同时进入同一房间；
- A 出价，B 能实时看到价格变化；
- 主播揭示卡片，所有观众实时看到；
- WebSocket 断开后不会影响服务端状态；
- 重新连接可拿到最新快照。

---

## 2.5 前端观众端负责人

### 负责内容

负责直播间观众体验。

前端模块位置：

```text
src/pages/live-room/
src/features/auction/
src/features/ranking/
src/features/playbook/
src/features/realtime/
```

### 页面能力

- 展示当前拍品；
- 展示当前价；
- 展示下一口价；
- 展示倒计时；
- 出价按钮；
- 自定义金额出价；
- 实时排行榜；
- 信任揭示卡片；
- Duel Mode 展示。

### 验收标准

- 能连接真实后端；
- 不使用 mock 服务；
- 出价后 UI 实时更新；
- 排行榜实时更新；
- 信任卡片揭示后实时展示；
- Duel Mode 有明显视觉状态。

---

## 2.6 前端主播端负责人

### 负责内容

负责主播/运营控制台。

前端模块位置：

```text
src/pages/host-console/
src/features/auction-admin/
src/features/trust-card-admin/
src/features/settlement/
```

### 页面能力

- 创建拍品；
- 设置起拍价；
- 设置最低加价；
- 设置竞拍时长；
- 配置信任揭示卡片；
- 点击开拍；
- 点击揭示卡片；
- 点击进入 Duel Mode；
- 点击落锤成交。

### 验收标准

- 主播端可以完整驱动一场 V1 竞拍；
- 所有操作都走真实后端接口；
- 操作后观众端能实时收到变化。

---

## 2.7 测试与答辩负责人

### 负责内容

- 写测试用例；
- 跑通演示脚本；
- 整理答辩材料；
- 记录系统亮点；
- 准备风险与扩展回答。

### 主要产出

```text
docs/openclaw/v1/06-测试用例.md
docs/openclaw/v1/07-演示脚本.md
docs/openclaw/v1/08-答辩要点.md
```

### 演示主线

```text
主播创建拍品
 ↓
开拍
 ↓
两个观众同时出价
 ↓
排行榜变化
 ↓
主播揭示证书/瑕疵卡片
 ↓
最后阶段进入 Duel Mode
 ↓
落锤成交
```

### 答辩亮点

- 不是普通竞拍页，是直播互动竞拍玩法系统；
- 出价链路服务端权威；
- WebSocket 实时广播；
- 状态机保证竞拍规则；
- 信任揭示解决非标品交易顾虑；
- Duel Mode 强化短视频直播竞技感；
- AI 作为旁路玩法助手，不阻塞交易主链路。

---

## 3. 推荐 GitHub 协作方式

### 3.1 分支规则

建议统一：

```text
main                         稳定主分支
feature/proto-v1             proto 契约
feature/biz-auction          后端交易核心
feature/realtime-ws          后端实时链路
feature/frontend-live-room   前端观众端
feature/frontend-host        前端主播端
feature/docs-demo            测试/演示/答辩文档
```

### 3.2 PR 规则

每个 PR 必须写清楚：

```text
做了什么
影响哪些模块
怎么验证
还有什么没做
```

### 3.3 禁止事项

- 不直接往 main 堆大改；
- 不把 mock 服务放进前端仓库；
- 不绕过 proto 私自改前端 DTO；
- 不在交易核心链路里阻塞调用 AI；
- 不在没讨论清楚前扩展支付、订单、保证金。

## 4. 第一阶段任务拆分

第一阶段目标：把 V1 契约和系统设计定下来。

### Task 1：需求确认

负责人：产品需求负责人

产出：

```text
docs/openclaw/v1/01-需求草案.md
```

验收：团队确认 V1 做什么、不做什么。

### Task 2：系统设计

负责人：系统架构负责人

产出：

```text
docs/openclaw/v1/02-系统设计构思.md
```

验收：模块边界、数据流、事件流讲清楚。

### Task 3：proto 契约

负责人：系统架构负责人 + 后端交易核心负责人

产出：

```text
api/auction/service/v1/auction.proto
```

验收：前后端都能看懂接口和数据结构。

### Task 4：前端页面拆分方案

负责人：前端观众端负责人 + 前端主播端负责人

产出：

```text
frontend docs 或 issue
```

验收：观众端和主播端页面职责清楚。

### Task 5：演示脚本草案

负责人：测试与答辩负责人

产出：

```text
docs/openclaw/v1/07-演示脚本.md
```

验收：团队知道最后要演示什么。

## 5. 建议的人力优先级

如果团队只有 3 个人：

```text
A：产品 + 架构 + proto
B：后端交易核心 + 实时链路
C：前端观众端 + 主播端 + 演示
```

如果团队有 4 个人：

```text
A：产品 + 架构 + proto
B：后端交易核心
C：后端实时链路 + 数据层
D：前端观众端 + 主播端
```

如果团队有 5-6 个人：

```text
A：产品/答辩
B：架构/proto
C：后端交易核心
D：后端实时链路/数据层
E：前端观众端
F：前端主播端/演示
```

## 6. 我的建议

当前最关键的不是马上写代码，而是先把三件事定住：

1. V1 需求边界；
2. 系统架构表达；
3. proto 契约。

这三件事定了之后，前后端才能并行，不然多人协作很容易互相等、互相改、互相推翻。
