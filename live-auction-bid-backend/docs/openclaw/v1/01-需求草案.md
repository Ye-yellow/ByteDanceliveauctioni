# V1 需求草案：短视频直播互动竞拍系统

> 状态：半成品草案，用于讨论，不作为最终需求冻结版。  
> 方式：按 gstack office-hours 思路先澄清问题和范围，再决定实现。  
> 文档位置：`docs/openclaw/v1/01-需求草案.md`

## 1. 赛题背景理解

赛题是一个“电商系统”方向，但它不是普通商城，也不是简单拍卖系统。

题目强调的是：

```text
短视频时代交易新玩法：抖音电商直播竞拍全栈系统设计
```

核心场景：

- 直播电商；
- 珠宝、二手奢侈品等高价值非标品；
- 商品难以统一定价；
- 通过动态竞拍提升互动感和成交价值；
- 要覆盖从商品上架、规则设定、实时出价、动态排名到落锤成交的闭环；
- 技术上重点考 WebSocket 实时通信、状态一致性、高并发；
- AI 是加分创新点，但不应该喧宾夺主。

## 2. 我们对项目的初步定位

不做普通“竞拍页面”。

我们要做的是：

> 面向短视频直播电商的互动竞拍玩法系统。

核心不只是“谁出价最高”，而是把直播间的互动、围观、信任建立和最后争夺，变成可配置、可追踪、可实时广播的竞拍机制。

第一版先不要过度追求大而全，重点做清楚：

```text
一个直播间
一个拍品
一套竞拍规则
多人实时出价
实时排名变化
最后落锤成交
玩法事件辅助互动
```

## 3. 目标用户

### 3.1 主播 / 商家 / 运营

他们需要：

- 上架竞拍商品；
- 设置起拍价、加价幅度、竞拍时间；
- 控制开拍、暂停、落锤；
- 看到实时出价和排名；
- 通过玩法机制提升直播间参与度；
- 在非标品场景下更好地建立用户信任。

### 3.2 观众 / 买家

他们需要：

- 看到当前竞拍商品；
- 看到当前价、下一口价、剩余时间；
- 实时出价；
- 看到自己和其他人的排名变化；
- 在下单前获得足够商品信息，比如证书、瑕疵、细节、售后说明；
- 感受到直播竞拍的紧张感和互动感。

### 3.3 平台 / 系统方

系统需要：

- 保证出价顺序和状态一致；
- 防止最后一秒偷塔造成争议；
- 记录可追溯竞拍事件；
- 为后续订单、支付、风控留下扩展空间。

## 4. V1 范围

### 4.1 V1 必须有

#### A. 商品上架

主播/运营可以创建竞拍商品，最少包含：

- 商品标题；
- 商品描述；
- 商品图片；
- 起拍价；
- 最低加价幅度；
- 竞拍时长。

#### B. 竞拍规则设置

V1 支持基础规则：

- 起拍价；
- 最低加价幅度；
- 倒计时；
- 最后出价自动延时，防止最后一秒偷塔；
- 每个用户出价必须高于当前价 + 最低加价幅度。

#### C. 实时出价

观众可以在直播间出价。

系统需要：

- 校验出价是否合法；
- 更新当前最高价；
- 更新当前领先用户；
- 广播给直播间所有连接用户。

#### D. 实时排名

系统维护当前竞拍排名。

V1 排名可先简单：

```text
按每个用户的最高出价排序
```

#### E. 落锤成交

倒计时结束后：

- 锁定最高价；
- 锁定最高出价用户；
- 生成成交结果；
- 拍品状态变为已成交。

#### F. 基础玩法事件

V1 先不做复杂 AI。

但要预留玩法事件概念：

- 信任揭示事件；
- 群体共振事件；
- 双人巅峰竞拍事件。

第一版可以先定义模型和展示，不一定全自动触发。

## 5. V1 暂不做

为了避免一开始做散，V1 暂不做：

- 真实支付；
- 保证金；
- 复杂订单履约；
- 多直播间大规模管理；
- 用户账号体系；
- 真正 AI 自动决策；
- 复杂风控；
- 多商品组合拍；
- 盲盒类玩法。

这些可以作为 V2/V3。

## 6. 玩法方向初稿

我们不主打“AI 定价”。

AI 定价风险大，而且不够贴合抖音直播场景。

更好的方向是：

> 玩法赋能。

也就是系统围绕直播竞拍设计新的互动玩法。

### 6.1 信任揭示竞拍

非标品最大问题是信任。

用户可能不是不想买，而是不敢拍。

玩法：

- 竞拍过程中逐步展示商品信息；
- 比如证书、瑕疵、细节图、售后承诺；
- 主播端可以手动触发；
- 后续可由系统根据评论关键词和出价转化率推荐触发。

V1 可以先做成：

```text
主播手动点击“展示证书/展示瑕疵/展示细节”
观众端实时看到对应信任卡片
```

### 6.2 群体共振竞拍

直播间大量用户只是围观，不一定出价。

玩法：

- 围观用户点赞、评论、投票可以触发信息揭示或互动事件；
- 但不能直接改变成交价，避免不公平。

V1 可以先做成：

```text
显示群体热度进度条
达到阈值后解锁一个商品信息卡
```

### 6.3 双人巅峰竞拍

最后阶段如果前两名反复竞争，系统进入 Duel Mode。

玩法：

- 前端突出展示 Top2 竞争；
- 倒计时进入更紧张的短轮次；
- 最后出价触发反狙击延时；
- 其他观众可以围观助威。

V1 可以先做成：

```text
当 Top2 连续交替出价达到阈值，前端展示“双人巅峰竞拍”状态
```

## 7. AI 的位置

V1 不做 AI 自动决策。

AI 在本项目中的长期定位：

> 直播竞拍玩法助手。

它未来可以做：

- 判断当前是否信任阻塞；
- 建议主播展示哪类商品信息；
- 判断是否适合进入 Duel Mode；
- 生成主播控场提示；
- 拍后复盘玩法效果。

但 V1 应该先通过规则引擎跑通，不急着接大模型。

## 8. 技术需求初稿

### 8.1 前端

前端至少两个视角：

#### 观众端

- 商品展示；
- 当前价；
- 下一口价；
- 倒计时；
- 出价按钮；
- 实时排名；
- 信任揭示卡片；
- Duel Mode 展示。

#### 主播/运营端

- 创建拍品；
- 设置竞拍规则；
- 开拍；
- 手动落锤；
- 手动触发信任揭示卡片；
- 查看当前排名和出价记录。

### 8.2 后端

后端采用 Go + Kratos 风格。

核心模块：

```text
api/                 协议契约
app/auction/service  竞拍服务
internal/server      HTTP/WebSocket
internal/service     请求适配
internal/biz         竞拍业务规则
internal/data        数据存储
internal/realtime    房间广播
```

### 8.3 实时通信

使用 WebSocket。

主要事件：

```text
lot.updated
bid.accepted
ranking.updated
lot.settled
trust.revealed
crowd.updated
duel.started
duel.ended
```

### 8.4 状态一致性

V1 可以先用单进程内存实现，便于快速跑通。

但设计上必须给 Redis 原子出价留位置。

后续升级：

```text
Redis Lua 原子校验出价
Redis ZSET 排名
Redis Stream 竞拍事件
MySQL 持久化拍品和成交记录
```

## 9. 第一版最小闭环

我建议第一版只做这个闭环：

```text
主播创建拍品
 ↓
设置起拍价和加价幅度
 ↓
开拍
 ↓
观众进入直播间
 ↓
多人实时出价
 ↓
排行榜更新
 ↓
主播触发信任揭示卡片
 ↓
最后阶段触发 Duel Mode 展示
 ↓
落锤成交
```

如果这个闭环能跑通，比赛演示就有基础了。

## 10. 需要你确认的问题

这份需求还没定稿，我需要你确认下面几个问题：

1. V1 是否只做单直播间，还是一开始就支持多直播间？
2. V1 是否需要登录用户，还是先用游客昵称？
3. 主播端是否必须做，还是先通过接口创建拍品？
4. 信任揭示卡片是否作为 V1 核心功能？
5. Duel Mode 是否作为 V1 演示重点？
6. 群体共振是否要接真实点赞/评论，还是先用前端按钮模拟互动事件？
7. 是否坚持 V1 不接 AI，只保留 AI 玩法助手位置？

## 11. 我对 V1 的建议

我的建议是：

- 单直播间；
- 游客昵称；
- 做简单主播端；
- 信任揭示作为核心；
- Duel Mode 作为演示亮点；
- 群体共振先做进度条，不接真实抖音数据；
- AI 暂不接模型，只保留玩法助手设计。

这样范围可控，也足够贴合赛题。
