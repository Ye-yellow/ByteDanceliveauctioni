# 09-技术栈选型方案

> 状态：待确认  
> 目的：在继续改代码前，先把 liveauction V1 需要的技术栈、边界和替换规则定清楚。  
> 原则：不写 demo，不写 mock；每次开发都按系统开发标准推进。

## 1. 总原则

### 1.1 先选型，再实现

如果当前项目缺少某类技术栈，不能直接写临时 demo 代码。

正确流程：

```text
发现能力缺口
  ↓
提出 2-3 个技术栈方案
  ↓
给出推荐方案和理由
  ↓
用户确认
  ↓
落文档 / 设计
  ↓
实现
```

例如：需要持久化数据时，不能继续用内存或手写 SQL demo；应该先明确数据库技术栈，再实现。

### 1.2 有替代就删旧实现

如果新技术栈已经替代旧实现，旧实现不要长期并存。

规则：

- MySQL + Redis + GORM 已落地主路径，`MemoryStore` 已删除；
- GORM data repo 已落地主路径，手写 `database/sql` repo 和 `schema.go` 已删除；
- 如果后续引入正式 migration 工具，启动时自动建表应删除；
- 只保留清晰的一条主路径，避免“demo 路径 + 正式路径”混在一起。

### 1.3 禁止 demo / mock

本项目禁止：

- 前端 mock 服务；
- 前端模拟数据；
- 后端内存 demo 作为默认路径；
- 为了快速跑通而写无法进入正式系统的临时实现。

允许：

- 单元测试里的测试数据；
- 明确隔离在 `test/` 下的测试辅助；
- 文档中的设计草案。

## 3. 后端技术栈

### 3.1 语言与服务框架

推荐：

| 能力 | 技术栈 | 说明 |
| --- | --- | --- |
| 语言 | Go | 主语言，适合高并发服务和系统设计表达 |
| 服务框架 | Kratos | 匹配 proto 契约源、HTTP/gRPC、分层风格 |
| API 契约 | Protocol Buffers | 后端、前端、文档统一契约源 |
| HTTP 映射 | google.api.http + Kratos transport/http | 基于 proto 生成 HTTP 路由 |

结论：继续使用 Go + Kratos + Proto。

### 3.2 后端分层

保持当前 Odin/Kratos 风格：

```text
api/auction/service/v1        proto 契约
app/auction/service/cmd       启动入口
app/auction/service/internal/server
app/auction/service/internal/service
app/auction/service/internal/biz
app/auction/service/internal/data
app/auction/service/internal/realtime
app/auction/service/internal/pkg
```

职责边界：

| 层 | 责任 |
| --- | --- |
| api | 契约源，不写业务实现 |
| server | HTTP/WebSocket server 注册 |
| service | proto req -> usecase -> reply |
| biz | 业务状态机、竞拍规则、玩法规则 |
| data | MySQL / Redis / GORM / cache / transaction |
| realtime | WebSocket hub、广播、连接管理 |
| pkg | ID、clock 等通用基础能力 |

## 4. 数据库与 ORM

### 4.1 MySQL

推荐：MySQL 8.x

用途：

- 拍品聚合；
- 竞拍规则；
- 信任揭示卡片；
- 出价流水；
- 成交结果；
- 事件日志。

原因：

- 适合结构化交易数据；
- 能体现系统设计中的事务、索引、状态查询；
- 和 Go/GORM 生态成熟；
- 适合比赛答辩表达。

### 4.2 Redis

推荐：Redis 7.x

用途：

- 出价幂等键；
- 高频排行榜；
- 房间快照缓存；
- 后续事件流 / PubSub / Stream。

V1 优先级：

1. 幂等键；
2. 排行榜 ZSET；
3. 快照缓存；
4. Redis Stream。

### 4.3 ORM

推荐：GORM

原因：

- Go 项目常用；
- 符合用户要求；
- data/ref 中也是通过 `gormDB.Raw(...)` / `gormDB.Exec(...)` 的方式写 data 层；
- 可以同时支持 model struct、Raw SQL、事务。

使用方式：

```text
简单 CRUD：GORM Model / Where / Save
复杂 SQL：gormDB.Raw / gormDB.Exec
事务：db.Transaction 或 tx 注入 context
```

不推荐继续：

- 默认 `MemoryStore`；
- 直接在 repo 里大量使用 `database/sql`；
- 启动时到处散落 schema SQL。

## 5. 数据迁移工具

需要选择。

### 方案 A：GORM AutoMigrate

优点：

- 快；
- 和 GORM 模型一致；
- V1 够用。

缺点：

- 对生产级迁移可控性弱；
- 不适合复杂变更审计。

### 方案 B：goose / golang-migrate

优点：

- 正式 migration 文件；
- 可回滚；
- 更像真实工程。

缺点：

- V1 多一层工具成本。

### 推荐

V1 推荐：

```text
GORM AutoMigrate + 文档化表设计
```

后续如果要进一步工程化，再切到 `goose` 或 `golang-migrate`。

但无论选哪个，都不要手写散落 schema 作为主路径。

## 6. 后端 data 层目标形态

参考 `internal/data/ref/a.go`，data 层应该像这样：

```text
repo 持有 gorm.DB / redis.Client / log
方法内部通过 tryExtractTx(ctx) 拿 tx 或 db
GORM Raw / Exec / Model 操作数据库
Redis cache 失败不影响主流程时记录 warn
业务错误返回英文
```

推荐结构：

```text
internal/data/
  data.go              组装 DB / Redis / repo
  model.go             GORM model struct
  lot_repo.go          LotRepository 实现
  bid_repo.go          BidRepository 实现
  event_repo.go        事件日志实现
  cache.go             Redis key/cache 方法
  transaction.go       事务上下文能力
```

要删除或替换：

```text
memory_store.go        删除
clone.go               如果只服务 MemoryStore，则删除
store.go               改为 GORM Store/Data，不保留 database/sql 主路径
schema.go              删除，改为 GORM AutoMigrate 或 migration 工具
sql_lot_repo.go        改为 gorm 风格 lot_repo.go
sql_bid_repo.go        改为 gorm 风格 bid_repo.go
```

## 13. 服务注册与发现

用户已确认：服务注册与发现选择 Consul。

定位：

- V1 单服务阶段可以先通过 Docker Compose service name 和环境变量直连；
- 但系统设计和后续多服务拆分以 Consul 作为服务注册发现中心；
- Kratos 服务注册应预留 Consul registry 接入；
- 后续如果拆分 `auction-service`、`realtime-gateway`、`room-service`、`order-service`、`ai-playbook-service`，统一注册到 Consul。

建议依赖：

```text
github.com/go-kratos/kratos/contrib/registry/consul/v2
github.com/hashicorp/consul/api
```

Docker Compose 后续应增加：

```text
consul
```

## 14. 下一步建议

如果确认这套技术栈，下一步应该按顺序做：

1. 删除 `MemoryStore`；
2. 删除 `schema.go` 和手写 `database/sql` 主路径；
3. 引入 GORM；
4. 建立 GORM model；
5. 用 GORM 重写 data repo；
6. Redis 保留幂等能力；
7. 跑 `go test ./...` 和 `go build`；
8. 再提交。

不要再继续在现有 `database/sql + schema.go` 上堆实现。
