# 抖音电商 AI 全栈课题需求对齐实施方案

> 需求基线：用户 2026-05-20 提供的《抖音电商AI全栈课题-直播竞拍全栈系统（宣讲版）》PDF。
>
> 后续 liveauction 不再按自由发挥的直播竞拍产品推进，而是围绕该课题评分点落地。已有工程规则继续生效：不写 mock、不保留被替代实现、业务错误统一 `reply.result`、测试集中在 `app/auction/service/test`、每次基础设施/设计变更都写工程思考和设计模式。

## 1. 课题核心目标

构建一套直播竞拍全栈系统，实现从商家发布竞拍到用户实时出价、动态排名、成交生成订单的完整闭环。

主流程必须闭合：

```text
商品上架
  -> 规则配置
  -> 主播开拍
  -> 用户实时出价
  -> WebSocket 同步价格/排名/倒计时
  -> 自动延时 / 封顶价自动成交 / 主播异常取消
  -> 竞拍结束
  -> 自动生成订单
  -> 用户查看结果 / 模拟支付
```


## 1.1 产品端划分：用户 H5/小程序端 与 商家主播后台分离

根据课题闭环，系统不是一个单一后台页面，而应分成两个产品端：

1. **用户 H5 竞拍端（买家侧 / 移动端）**
   - 面向用户/买家；
   - 承接“用户实时出价”“WebSocket 同步价格/排名/倒计时”“用户查看结果/模拟支付”“移动端 H5 体验”；
   - 形态优先为 mobile-first H5，后续可扩展为小程序端；
   - 小程序不是当前后台的一部分，应作为独立端复用后端 API/WebSocket。

2. **商家 / 主播 / 运营后台（管理侧 / PC Web）**
   - 面向商家、主播、运营、管理员；
   - 承接“商品上架”“规则配置”“主播开拍”“异常取消”“订单管理”“数据看板”；
   - 商家端与主播端当前可以合并在同一后台，通过角色和权限区分能力，不拆成两个入口堆概念。

设计原则：用户 H5/小程序竞拍端必须与商家主播后台分离。首页和答辩材料应表达为“用户 H5 竞拍端 + 商家主播后台”两端闭环，避免把 H5/小程序体验误归入运营后台。

## 3. 对照课题要求的缺口

| 课题要求 | 当前状态 | 处理策略 |
|---|---:|---|
| 0 元起拍 | 部分支持但未显式设计/测试 | BidRule 明确允许 start_price=0，补测试和 UI 提示 |
| 固定加价幅度 | 已支持 min_increment | 保留，补拒绝原因和前端提示 |

## 4. 工程设计原则

### 4.1 Auction Domain State Machine

竞拍状态流转应集中在 `internal/biz/auction`：

```text
DRAFT -> LIVE -> SETTLED
              -> CANCELLED
```

规则：

- 只有 DRAFT 可修改规则；
- 只有 LIVE 可出价、延时、取消、落锤；
- 达到封顶价直接 SETTLED；
- 取消必须记录 cancel_reason；
- 成交必须生成订单。

设计模式：**Domain State Machine + Usecase Orchestration**。业务状态机留在 biz，不把规则散落到 service/data/server。

### 4.2 Repository + Unit of Work

`data` 层继续持有 GORM/Redis/事务/cache/outbox。新增订单后，封顶成交或落锤成交必须在同一个 MySQL transaction 内完成：

```text
update lot settled
insert bid if applicable
insert order
insert auction_events
commit
publish Redis Stream / WebSocket
```

设计模式：**Repository + Unit of Work + Transactional Outbox**。

### 4.3 Result Envelope

所有业务拒绝都包装进 reply.result 或业务 reply 字段：

- 出价不足：`accepted=false + reject_reason`；
- 版本冲突：`result.code=409001`；
- 未登录/无权限：`401001/403001`；
- 不走 `reply + error` 双语义。

设计模式：**Result Envelope Adapter**。

### 4.4 Realtime Recovery

前端 WebSocket 不能只连一次。需要：

```text
connected -> heartbeat missed -> reconnecting -> connected
                              -> failed
```

每次重连成功后调用 `GetRoomSnapshot` 恢复权威状态，避免本地事件丢失。

设计模式：**Reconnect State Machine + Snapshot Recovery**。

## 6. 下一步实施顺序

建议下一轮从后端 P0 开始：

```text
1. Proto 增加 cap_price / CancelLot / OrderService 或 Order RPC
2. data 增加 order model + schema + repo
3. biz 增加封顶成交、取消、订单生成状态机
4. service 统一 reply.result 包装
5. test 目录补状态机与事务测试
6. 前端同步契约，补 UI 和提醒
7. npm/go/docker compose 验证
```

原因：封顶价、取消、订单是课题硬要求，必须先让后端权威状态闭环，再做前端体验。

## 7. CancelLot 异常取消落地说明（2026-05-20）

本轮按“先闭合一个端到端功能”的原则，仅实现主播异常取消竞拍 `CancelLot`，暂停封顶价和订单聚合扩展，避免一次变更跨过多个业务边界。

### 7.1 工程思考

- **契约先行**：在 `auction.proto` 增加 `CancelLotRequest/CancelLotReply` 和 `/api/lots/{lot_id}/cancel` HTTP 路由；`Lot` 系统化新增 `cancel_reason/cancelled_at_unix_ms`，使取消原因成为权威状态的一部分，而不是只存在日志或事件里。
- **状态机收口**：biz 层 `CancelLot` 只允许 `LIVE -> CANCELLED`，拒绝 DRAFT/SETTLED/CANCELLED 再取消；取消后禁止继续出价，并关闭 Duel 展示状态。
- **事务边界**：data 层沿用既有 `LotRepository.Save`，在同一个 MySQL transaction 内保存 lot 状态和 `LOT_CANCELLED` event；事务提交后再写 Redis Stream，保持现有 Transactional Outbox 语义。
- **响应语义**：service 层不返回业务 `error`，统一通过 `reply.result` 表达权限、状态不合法、参数缺失等可预期拒绝。
- **测试隔离**：新增测试仍集中在 `app/auction/service/test`，覆盖状态机、持久化和事件发布，不污染 internal 实现目录。

### 7.2 设计模式

- **Domain State Machine**：取消规则由 `internal/biz/auction` 统一执行，避免 service/data 分散判断状态流转。
- **Repository + Unit of Work**：复用 `LotRepository.Save(ctx, lot, expectedVersion, events)`，把聚合状态和事件写入同一事务。
- **Result Envelope Adapter**：service 层把 biz 错误适配为 `ReplyResult`，前端只需解析统一 envelope。
