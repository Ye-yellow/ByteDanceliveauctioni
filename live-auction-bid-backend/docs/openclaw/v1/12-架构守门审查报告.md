# 架构守门审查报告

> 审查时间：2026-05-20 20:10+08
> 审查范围：`PROJECT_CHARTER.md`、`docs/openclaw/v1/10-抖音课题需求对齐实施方案.md`、后端 README、`docs/openclaw/v1` 关键文档、当前后端实现、兄弟前端仓库关键链路。
> 约束：本次只做架构守门与文档输出；不 commit、不 push；不抢 backend P0 业务代码。

## 1. 总体结论

当前路线已经从“普通 demo”被拉回到课题主线，以下底座方向是对的：

- 后端主路径已明确为 Go + Kratos + Proto + MySQL/GORM + Redis + Consul；
- data/server/biz/service/realtime 分层基本成型，旧 `MemoryStore` / `database/sql` 主路径未在生产实现中复活；
- service 层当前多数业务错误已收敛到 `reply.result`；
- 测试文件当前集中在 `app/auction/service/test`，未发现实现目录下新增 `*_test.go`；
- 前端已建立 Result Envelope Adapter、Auth Token Adapter、Realtime Normalizer 等防腐层。

但课题 P0 闭环仍未完成，且有几处“文档/契约仍停留在旧 V1 简化口径”的风险。最高优先级应继续围绕：封顶价自动成交、异常取消、成交订单、WebSocket 心跳重连快照恢复、移动 H5 提醒闭环推进。

## 2. 必须修（阻断课题 P0 闭环）

### M1. Proto 契约仍缺封顶价、取消、订单/模拟支付主路径

证据：

- `api/auction/service/v1/auction.proto` 的 `BidRule` 只有 `start_price/min_increment/duration/anti_snipe`，没有 `cap_price`；
- `AuctionService` 暂无 `CancelLot` RPC；
- 响应/模型中暂无订单聚合、订单查询、模拟支付接口；
- `SettleLotReply` 只返回 lot/event，无法表达成交后自动生成订单。

影响：这直接缺失课题 P0：封顶价自动成交、主播异常取消、成交后订单、用户结果/模拟支付。

守门建议：backend P0 负责人应先改契约，再实现 data/biz/service/frontend 同步。不要绕过 proto 在前端或 HTTP handler 里临时补字段。

### M2. 业务状态机尚未覆盖封顶成交、取消和订单事务

证据：

- `AcceptBid` 只更新当前价、领先者、自动延时和版本，没有检查 `cap_price` 并自动 `SETTLED`；
- `SettleLot` 只锁定 lot 成交字段，没有在同一事务内生成 order；
- `LOT_STATUS_CANCELLED` 枚举已存在，但未发现对应业务函数/RPC/数据字段 `cancel_reason` 主路径。

影响：当前状态机仍是 `DRAFT -> LIVE -> SETTLED` 为主，和课题要求的 `DRAFT -> LIVE -> SETTLED/CANCELLED` + 成交订单闭环不一致。

守门建议：状态机规则必须留在 `internal/biz/auction`，订单生成的事务边界在 data 层 Unit of Work 中完成；不要在 service/server 层拼规则。

### M3. WebSocket 重连/心跳/快照恢复仍不达 P0

证据：前端 `useAuctionRoom.ts` 当前只在 mount 时拉一次 snapshot，然后建立一次 WebSocket；`onclose` 仅设置 disconnected，没有重连状态机、退避、心跳检测或重连成功后强制恢复 snapshot。

影响：课题明确要求 WebSocket 心跳保活、自动重连、断线后快照恢复；当前只能算“能连一次”，不满足答辩硬点。

守门建议：前端应实现 `connected -> heartbeat missed -> reconnecting -> connected/failed`，每次 reconnect open 后调用 `GetRoomSnapshot` 覆盖本地状态；后端同时明确 ping/pong 或 app-level heartbeat 策略。

### M4. 自动延时没有独立事件/提醒语义

证据：`AcceptBid` 已在窗口内修改 `EndsAtUnixMs`，但当前事件只提交 `BID_ACCEPTED` / `RANKING_UPDATED` / 可选 `DUEL_STARTED`，没有独立的“延时触发”事件或 reason；前端也只对 `BID_REJECTED` 设置 notice。

影响：自动延时虽然可能改变倒计时，但答辩和用户体验很难证明“延时提醒”已实现。

守门建议：补 `AUCTION_EXTENDED` 或在 `LOT_UPDATED` 中明确 `reason=ANTI_SNIPE_EXTENDED`，前端加入 feed/toast 展示。

### M5. 旧 V1 简化口径仍残留在 proto 注释中

证据：`auction.proto` 顶部仍写着：

- “V1 暂不处理支付、订单履约、保证金”；
- “V1 可以先使用单进程内存存储，后续再替换 Redis/MySQL”。

影响：这与 `PROJECT_CHARTER.md` 的最高纲领冲突，会误导后续 agent/开发者重新走 demo/mock/简化路线。

守门建议：在 backend P0 契约修改时一并清理 proto 注释；至少要把“内存存储”和“不处理订单/支付”改为“本课题 P0 要处理成交订单与模拟支付，数据主路径为 MySQL/Redis”。

## 3. 应该修（不一定阻断单点开发，但会扩大偏差）

### S1. 0 元起拍缺显式测试与 UI 表达

当前 `NewLotFromRequest` 未禁止 `start_price.amount=0`，但测试样例仍基本是 10000 起拍，缺少“0 元起拍可创建、可开拍、首笔合法出价为 min_increment”的明确用例。

建议：补 `app/auction/service/test` 下外部测试；前端创建/展示规则明确“0 元起拍”。

### S2. `reply.result` 基本统一，但需防止系统错误信息外泄

当前 service 返回形态基本正确：业务错误返回 `Reply{Result: ErrorResult(err)}, nil`。但 `ErrorResult` 的兜底 `500000` 直接返回 `err.Error()`，如果底层 MySQL/Redis/JWT 错误进入此处，可能把内部细节暴露给前端。

建议：业务错误保持具体；未知系统错误统一文案为“internal error”，详细错误留日志/trace。

### S3. 订单/取消引入时必须更新数据库设计文档和 schema

`08-数据库表设计.md` 当前只覆盖 lot/bid/event/user 之前的主线，没有订单表、取消原因、封顶价字段。后续如果只改 GORM model，不改文档和 `deploy/mysql/schema.sql`，会破坏“基础设施变更必须写设计模式/边界”的铁律。

建议：新增订单表设计、索引、状态机；lot 表增加 `cap_price_*`、`cancel_reason` 等字段时同步 docs/schema。

### S4. 前端 README 能力描述仍有“AI 气氛官”等非课题 P0 口径

前端 README 已声明课题 P0，但“能力”部分仍突出 AI 气氛官、响应式比赛演示页面；这不是当前硬闭环重点。

建议：把能力描述改为移动 H5 直播间、实时竞价、排名/倒计时、提醒 feed、订单/模拟支付；AI/气氛官放后置加分项。

### S5. Consul/MySQL/Redis 基础设施设计说明已有，但订单/支付基础设施模式还未沉淀

现有 README/05/08/09 已说明 Result Envelope、Repository + Unit of Work、Transactional Outbox、Registry + Health Check，这部分足够支撑已实现底座。新增订单/模拟支付时，还需要补“Order Aggregate + Payment Simulation Adapter”的设计说明，避免把支付按钮做成前端假状态。

## 4. 可后置（完成 P0 后再做）

- Redis ZSET 排行榜主路径：当前可从 MySQL 出价流水构建排名，P0 更急的是订单/取消/封顶；
- last_event_id 事件补偿回放：当前可先以 snapshot recovery 满足断线恢复，事件回放可作为稳定性加分；
- Duel/信任卡玩法深化：保留现有能力即可，避免抢占课题硬闭环资源；
- 正式 migration 工具替换 AutoMigrate：V1 可以继续 GORM AutoMigrate + 文档化表设计，答辩前再评估 goose/golang-migrate；
- AI 工具流程、状态机图、时序图、压测报告：实现闭环稳定后集中补答辩材料。

## 5. 分层边界审查

### 当前符合项

- `internal/service` 主要做 proto req/reply 适配和 result 包装；
- `internal/biz/auction` 承载状态机和竞价规则；
- `internal/data` 持有 GORM/Redis/事务/outbox；
- `internal/server` 持有 HTTP/WebSocket/health/registry；
- 前端把 result/auth/realtime normalization 放到 adapter 层，页面没有直接解析 Kratos 细节。

### 守门红线

后续 P0 实现不要突破以下边界：

- `CancelLot`、`cap_price`、订单状态机：只能进入 biz/usecase + data repo，不要写在 HTTP handler；
- 事务：封顶成交/落锤成交必须在同一 MySQL transaction 内更新 lot、插入 bid/order/event；
- WebSocket：server/realtime 只负责连接和广播，不写成交/取消业务规则；
- 前端：不能用本地 mock 订单或假支付状态冒充后端能力。

## 6. 替代实现残留审查

- 未发现生产主路径中恢复 `MemoryStore`、手写 `database/sql` repo 或 `schema.go` fallback；
- `database/sql` 命中只在 `internal/data/ref` 参考代码中，属于 ref 包；
- README 明确旧实现已删除；
- 需要清理的是 proto 注释中的“单进程内存存储”旧说法，而不是生产代码残留。

## 7. test 目录规则审查

- 当前未发现 `app/auction/service/internal/**` 下的 `*_test.go`；
- 当前测试集中在 `app/auction/service/test`，符合纲领；
- 后续 P0 新测试应继续放在该目录，不要恢复 `internal/biz/auction/lot_test.go` 这类旧文档建议。

## 8. 建议下一步执行顺序

1. 契约先行：`BidRule.cap_price`、`CancelLot`、`Order`/模拟支付相关 proto；
2. data：订单 model/schema/repo，lot 增加 cap/cancel 字段；
3. biz：封顶成交、异常取消、订单状态机、0 元起拍测试；
4. service：所有新增 RPC 继续统一 `reply.result`；
5. realtime：延时/成交/取消/被超越/领先事件语义补齐；
6. frontend：重连状态机 + snapshot recovery + mobile H5 feed + 订单/模拟支付页面；
7. docs：同步 08 数据库设计、README、proto 注释、评分点对照清单。
